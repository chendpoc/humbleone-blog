## 对使用者/维护者的判断

### 风险

- 如果只看版本号，会漏掉这种“同一版本说明继续修订”的重要更新
- 如果把 `0.8.61` 当成普通补丁版，会低估它对 multi-agent runtime、provider routing、goal lifecycle 和 plan/agent 边界语义的影响
- changelog 已经表现出“发版后仍持续增补”的特征，做升级评估时只存链接不存快照，会让回溯成本越来越高

### 机会

- 维护者正在把多 Agent 能力从“功能点罗列”推进到“运行时控制面说明更完整”的阶段
- 对依赖 sub-agent、goal mode、provider 切换、legacy app-server transport 的使用者来说，这种无新版本号的正文补写，本身就是高价值回归信号
- `0.8.61` 的补写越完整，越有利于后续把实际行为与 changelog 叙述逐项对照

## 建议动作

1. 继续把 changelog 哈希变化视为一级信号，不要只盯版本号。
2. 如果你在使用 CodeWhale 的 sub-agent / goal / provider routing 能力，把 `0.8.61` 当成一次仍在持续澄清语义的版本说明来复查。
3. 如果后续需要更精确回答“这一轮到底多了哪几句”，私有 state 需要增加完整 changelog 快照或分节摘要存档。
