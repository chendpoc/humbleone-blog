### `0.8.61` · 2026-06-14

**主题：** 多 Agent 工作的运行时控制面。TUI 在子 Agent 运行时保持响应，子 Agent 方向向 fleet-style durable workers 收敛，并引入按角色分配模型与 provider 的能力。

**需要关注：**

- WhaleFlow runtime foundations：worker runtime profile、跨 provider 模型注册表、provider readiness、context budget、provider adapter、resource telemetry。
- Per-role heterogeneous-model sub-agent routing：不同角色可以使用不同模型和 provider，例如 scout、synthesis、verifier 分流。
- Durable goal mode：跨轮目标进度、token/time 统计，以及 verifier-as-judge 完成门禁。
- 修复多子 Agent 启动导致 TUI 卡死的问题，是这版的关键稳定性改进。
- provider/model 路由隔离、context window 元数据、取消同步工具、MCP stdio proxy 启动提示等都属于运行时可靠性修复。

### `0.8.60` · 2026-06-13

**主题：** Agent Fleet 从模拟走向真实运行，以及只读 shell 并行能力。

**需要关注：**

- `codewhale fleet run` 切到 `codewhale exec --output-format stream-json` 的 durable worker 路径。
- 引擎支持保守的只读 shell 并行，但写入、stdin、后台 TTY、重定向、管道、命令替换等仍保持串行。
- WhaleFlow 支持声明式 JS/TS `workflow({...})` 编写形式，但只编译，不执行用户 JavaScript。
- 新增 GLM-5.2、Z.ai、StepFun / StepFlash、MiniMax 等模型和 provider 路由。
- `agent_eval` 默认不再阻塞，必要时通过 `block:true` 显式等待。

### `0.8.59` · 2026-06-12

**主题：** 模型/provider 扩展、交互体验改进、目标生命周期、线程目标 API，以及大量运行时与本地化改造。

**需要关注：**

- Moonshot/Kimi provider 默认模型更新到 `kimi-k2.7-code`，并加入 OpenRouter registry row。
- CLI 非交互启动默认进入 concise prompt/output discipline，交互式 TUI 仍保持 normal。
- 打开没有 instruction 文件的目录时，不再落盘 `.codewhale/instructions.md`，改为内存中的 bounded generated project overview。
- `/goal` 成为 session goal 的主要命令面，支持 pause、resume、complete、blocked、clear。
- App-server 侧新增 thread goal set/get/clear API，并记录 Codex-style 状态与 token/time accounting。
- 添加 active provider fallback chain、Hugging Face provider、provider metadata registry 等 provider 路由基础设施。
- 修复 SSE `data:{...}` 无空格格式、runtime thread detail N+1 reads、project-local hook trust boundary、skills sync latency 等问题。

### `0.8.58` · 2026-06-11

**主题：** 原生 Anthropic provider、Hooks v2、可点击 sidebar、`codewhale exec` 工具限制参数，以及 provider-aware sub-agent routing。

**需要关注：**

- 原生 Anthropic Messages API adapter 替代 OpenAI 方言 shim，支持 adaptive thinking、prompt-cache breakpoints、signed-thinking replay、cache usage telemetry、SSE error envelopes。
- Hooks v2 支持 JSON decision：`allow`、`deny`、`ask`，并支持输入重写和附加上下文。
- `codewhale exec` 增加 `--allowed-tools`、`--disallowed-tools`、`--max-turns`、`--append-system-prompt`。
- sub-agent routing 不再硬编码 DeepSeek 模型校验，改为 provider-aware。
- 修复 sub-agent 高负载下 TUI freeze，以及 Windows hook command quoting 导致 JSON decision 无法解析的问题。

## 当前重点观察

1. CodeWhale 正在把 sub-agent 从交互概念推进到更接近 durable worker / fleet runtime 的执行模型。
2. Provider 和模型路由正在快速解耦，后续兼容性重点会落在 provider metadata、route isolation、context budget 和 reasoning trace 处理上。
3. WhaleFlow 从规格和运行时两侧扩张，值得继续观察它会不会成为 CodeWhale 的长期自动化编排层。
4. 近期多次修复 TUI freeze、blocking eval、provider wait、background jobs、cancellable tools，说明多 Agent 并发与交互响应性是当前高优先级稳定性主题。

## 对使用者/维护者的建议动作

| 场景 | 建议 |
| --- | --- |
| 依赖子 Agent 或 fleet 能力 | 优先关注 `0.8.60` 到 `0.8.61` 的运行时语义变化，尤其是默认非阻塞、worker lifecycle、needs-input checkpoint、per-role model routing |
| 使用自定义 provider 或 OpenRouter / Z.ai / Moonshot / Anthropic | 关注 provider route isolation、registry metadata、reasoning trace、context budget 相关变化 |
| 使用 hooks | 重点阅读 `0.8.58` 的 Hooks v2 行为，尤其是 `deny > ask > allow` 的优先级和 Windows 命令传递修复 |
| 写自动化或工作流 | 继续观察 WhaleFlow 的 JS/TS authoring 与 runtime profile 方向 |
