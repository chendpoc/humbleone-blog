## 每个版本做了什么

### `Unreleased`

当前 `Unreleased` 仍以 `Retroactive credits` 为主，性质是对既有修复/贡献的补记，而不是一批新的功能上线。它传递的主要信号是：

- 官方仍在补全历史归因
- changelog 不是“发版后冻结”的文档
- 后续就算没有新版本号，顶部区块也仍可能继续变化

### `0.8.61`

`0.8.61` 仍然围绕多 Agent 运行时控制面展开，但当前正文比我们在 2026-06-16 公开记录时更完整。现在从 changelog 正文中可以明确看到的重点包括：

- WhaleFlow runtime foundations：worker runtime profiles、跨 provider 模型注册、provider readiness、context budget、provider adapter、resource telemetry
- Per-role heterogeneous-model sub-agent routing：不同角色可指定不同 model/provider
- Durable goal mode：跨轮目标进度、token/time accounting、完成前 verifier-as-judge gate
- Parent-visible worker interaction contract
- legacy app-server HTTP transport 上的 OpenAI-compatible `/v1/chat/completions` 端点

`Changed` 与 `Fixed` 的层面也继续变厚，重点不只是“功能有了”，而是“运行时语义被进一步说清”。当前正文明确强调或补充了这些方向：

- sub-agent 正在收敛到 fleet-style durable workers
- sidebar 展示真实 worker lifecycle，而不是硬编码 `running`
- 子 Agent 不再只靠隐藏式 child-model auto-routing，而是暴露 `model_strength` / `thinking` 控制
- shell 与 verifier 默认后台化，阻塞等待必须显式触发
- Plan mode 被强调为严格只读，并且不应再对 repo / URL / version / PR / issue / API surface / 本地路径这类请求偷懒跳过调查
- provider/model route 为 session-local
- `context_length_exceeded`、同步工具可取消、MCP stdio proxy、background shell recovery 等运行时稳定性问题继续被修补
- 输入与交互层面也补入了更多细节：oversized pasted text 可继续编辑、bare digit 键位行为修正、provider picker 改为中性字母序、Work sidebar 不再残留过期策略行
- provider/兼容性层面新增了 Kimi/Moonshot 空函数参数 schema 兼容、Novita 默认 `/openai/v1` 端点等修复
- 依赖安全层面写入了 `ws` 8.21.0 固定

## 新功能和能力变化

虽然没有新版本号，但从当前 changelog 仍能读出几条实质性方向：

1. CodeWhale 正在把多 Agent 能力从“可并发调用”推进为“有明确控制面、生命周期、路由语义和恢复语义”的系统。
2. `0.8.61` 不再只是一个运行时稳定性版本说明，而是在持续补齐 control plane 叙述。
3. legacy app-server transport 上的 OpenAI-compatible 端点，说明运行时与外部接入面之间的桥接能力也在被正式写入 changelog。
4. 许多“看起来像小修复”的条目，实际上都在收紧 Plan / Agent / Worker / Provider 之间的行为边界。

## 兼容性、迁移、破坏性变化或配置变化

从当前 changelog 能直接确认的兼容性/行为变化包括：

- Plan mode 的能力边界更严格，默认按只读理解更安全
- shell / verifier 的后台化默认值，意味着旧自动化如果依赖阻塞等待，需要复查预期
- provider/model route 改为 session-local，依赖隐式共享状态的用法应该重新验证
- `model_strength` / `thinking` 暴露给 sub-agent 路径后，调度与成本心智模型会变
- provider 兼容层继续变化，例如 Novita 默认端点与 Kimi/Moonshot schema 处理
- 依赖安全固定（`ws` 8.21.0）意味着维护者已把安全问题写入同一版本说明

没有看到必须人工执行的单独迁移步骤，但这不是“无感更新”。对重度使用者来说，应该把它当成一次运行时语义仍在收敛中的版本说明。

## 值得重点关注的风险或机会

### 风险

- 只按“是否发了新版本”跟踪，会漏掉当前这类重要变化
- 对 sub-agent、goal、provider routing 的旧认知，可能已经落后于当前 changelog 的行为定义
- 如果团队内部只保存“最新链接”，以后很难回答“这周到底改了什么”

### 机会

- 这是观察 CodeWhale 多 Agent runtime 真实成熟度的好窗口
- changelog 已经足够详细，适合据此整理一份“运行时能力/边界核对表”
- 继续盯 `0.8.61` 能帮助判断未来 `0.8.62+` 是延续 control plane，还是转向别的主题
